# Правила разработки Laravel (Simai Monorepo)

## Базовые принципы
- Держи патч минимальным и локальным.
- Валидируй входные данные через `FormRequest`/валидационные слои.
- Применяй авторизацию на уровне middleware + domain logic.
- Избегай N+1 и тяжелых запросов без индексов/кэша.
- Не смешивай инфраструктурные и бизнес-изменения в одном патче.

## Стандарты пакетов Simai
- Новый/измененный пакет должен вписываться в поток:
  - `composer require simai/<module>`
  - `php artisan simai:install`
- Для модульной установки используй команду `simai:install-<module>`:
  - без интерактива;
  - идемпотентно;
  - зарегистрировано в провайдере.
- Module config `config/simai_<module>.php` должен быть источником:
  - операций (`operations`);
  - доступов (`accesses`, `access_levels`);
  - меню и module metadata.

## Мультиязычность
- Поддерживай минимум `ru` и `en`.
- Держи одинаковые ключи в обеих локалях.
- Не хардкодь UI-строки в контроллерах/Blade.
- Регистрируй переводы через `loadTranslationsFrom(...)`.

## Доступы и архитектура `simai/access`
- Используй middleware по назначению:
  - `access` для operation-level проверки;
  - `access.token` для API токенов;
  - `access.entity` для EntityConfig-ориентированных CRUD действий.
- Операции определяй в конфиге модуля, не в виде разрозненных строк по коду.
- Для новых сущностей/действий закладывай негативные сценарии (401/403) в тесты.

## Admin/UI архитектура
- CRUD регистрируй через `Route::simaiCrud(...)` / `Route::simaiApiCrud(...)`.
- Используй `EntityConfig` как центральное место описания полей/фильтров/sort/access map.
- Для HTML используй `admin::layouts.*` и `admin::crud.*`.
- Новые виджеты и шаблоны:
  - добавляй только при реальной нехватке текущих пресетов;
  - делай переиспользуемыми;
  - документируй в runbook/overview.

## Внешние зависимости и безопасность поставки
- Не добавляй CDN и внешние script/style URL в пакетный UI-код.
- Не зашивай URL сторонних сервисов в код.
- Любую внешнюю интеграцию выноси в config/env и документируй fallback.

## Обязательные артефакты изменений
- `SPEC.md` при изменении поведения/контракта.
- `CHANGELOG.md` при пользовательских изменениях.
- Runbook/ADR для сложных архитектурных или эксплуатационных изменений.

## Тестовая политика
- Любая поведенческая правка должна сопровождаться тестами.
- Любой багфикс должен содержать regression test.
- Минимальный набор запуска:
  - `composer run validate:repo`
  - `composer run lint:changed`
  - `vendor/bin/phpunit packages/<module>/tests`
