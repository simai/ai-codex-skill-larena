# Правила разработки пакетов Simai (Laravel)

Этот файл фиксирует обязательные правила для новых модулей и изменений в существующих пакетах монорепы.

## 1) Минимум команд установки

Целевой путь установки для потребителя:
1. `composer require simai/<module>`
2. `php artisan simai:install`

Правила для нового пакета:
- Добавляй команду установки с сигнатурой `simai:install-<module>`.
- Делай команду идемпотентной и неинтерактивной.
- Регистрируй команду в сервис-провайдере пакета.
- Не требуй отдельного ручного запуска, если можно встроиться в общий `simai:install` поток.

Почему:
- `simai:install` в `simai/admin` автоматически ищет все команды `simai:install-*` и запускает их в порядке зависимостей пакетов из `vendor/simai/*/composer.json`.
- После этого он применяет module-конфиги `config/simai_*.php` и выполняет общий bootstrap меню/доступов.

### Дополнительные правила для `simai:install-<module>`
- Команда должна возвращать успех при повторном запуске (idempotent by design).
- Команда не должна дублировать глобальные шаги `simai:install`:
  - не запускать общий `migrate --force` "вслепую";
  - не дублировать `vendor:publish --tag=admin` и `--tag=simai-admin-config`.
- Команда должна выполнять только модуль-специфичные действия:
  - безопасный upsert/seed собственных сущностей;
  - подготовка локальных дефолтов/конфига;
  - мягкая проверка предусловий без падения всего install-потока.
- Любая ошибка в модульной установке должна быть диагностируемой:
  - короткое сообщение в консоль;
  - лог без секретов/PII;
  - предсказуемый код возврата.

## 2) Обязательный module config для интеграции

Для модулей, которые добавляют UI/доступы/операции, создавай `config/simai_<module>.php` с секциями:
- `module_info`
- `menu`
- `access_levels` (если нужны)
- `accesses`
- `operations`

Требования:
- Коды операций должны быть стабильными и согласованными с проверками в коде.
- Конфиг должен быть безопасен к повторному применению (upsert-логика).

## 3) Мультиязычность пакетов

Обязательные правила:
- Поддерживай минимум `resources/lang/ru/*` и `resources/lang/en/*`.
- Следи за паритетом ключей между языками.
- Используй `loadTranslationsFrom(...)` в сервис-провайдере.
- Не оставляй хардкод UI-строк в PHP/Blade, кроме технических исключений.

Рекомендации:
- Добавляй тест на наличие ключей (минимум smoke-уровень) для критичных экранов/сообщений.

## 4) Архитектура прав доступа через simai/access

Обязательные правила:
- Используй `Simai\Access\Services\AccessChecker` и стандартные middleware:
  - `access`
  - `access.token`
  - `access.entity`
- Для API маршрутов применяй `['api', 'access.token']`.
- Для CRUD-роутов опирайся на `EntityConfig` + ability map, чтобы проверки были единообразными.
- Операции доступа объявляй в module config (`operations`), а не размазывай хардкодом по контроллерам.

## 5) Роуты, сущности и админ-страницы через SimaiCrud

Базовый путь для CRUD:
- `Route::simaiCrud(...)` для UI.
- `Route::simaiApiCrud(...)` для API.

Практики:
- Явно задавай `->param('id')`.
- Ограничивай действия через `->only([...])`.
- Для доп. действий используй `->extras([...])`.
- Проставляй предсказуемые route names через `->names([...])`.
- Роуты должны регистрироваться через `RouteMacros` (при необходимости `RouteMacros::register()`).

EntityConfig правила:
- Поля `allowedFields`, `writable`, `filters`, `sort`, `includes` должны быть ограничивающими, не permissive.
- Ability/Access маппинг обязателен для действий create/update/delete/status/reorder/custom.

## 6) Шаблоны, тема и UI-драйверы admin

Обязательные правила:
- Для страниц используй базовые layout из `admin::layouts.*`.
- Для CRUD используй стандартные шаблоны `admin::crud.*`.
- Для рендера опирайся на `SimaiResponder` + `UiDriverRegistry` (preset-based).

Расширение темы:
- Если стандартного preset не хватает, добавляй новый шаблон/виджет в совместимом стиле `simai/admin`.
- Новые виджеты/шаблоны должны быть переиспользуемыми и документированными.
- Не дублируй существующие partials без причины.

## 7) Запрет CDN и жестких внешних ссылок

Обязательные правила:
- Не добавляй внешние CDN ссылки (`unpkg`, `cdnjs`, `jsdelivr`, `googleapis` и т.д.) в Blade/JS/CSS пакетов.
- Не добавляй жестко зашитые URL сторонних сервисов в код.
- Если внешняя интеграция неизбежна, выноси endpoint в config/env и документируй fallback поведение.

Цель:
- Повторяемая офлайн/закрытая установка без внешних зависимостей.

## 8) Логи и учет изменений

Обязательные правила:
- Логируй важные операции и ошибки без секретов и PII.
- Для пользовательских/бизнес-изменений обновляй `CHANGELOG.md`.
- Для изменения контракта обновляй `SPEC.md`.
- Для сложных эксплуатационных изменений добавляй runbook/ADR.

Для Docara задач:
- Синхронизируй `docs/DOCARA_ISSUE_LOG.md` и `docs/DOCARA_DEVELOPMENT_LOG.md`.

## 9) Усиленное тестирование ("больше тестов")

Минимум:
- Для любой поведенческой правки добавляй или обновляй тесты.
- Для багфикса добавляй регрессионный тест, который воспроизводит проблему.
- Для прав доступа добавляй минимум 1 негативный тест (forbidden/unauthorized).
- Для новых CRUD/API добавляй feature-тесты на happy path + validation + access.

Рекомендуемый запуск:
- `composer run validate:repo`
- `composer run lint:changed`
- `vendor/bin/phpunit packages/<module>/tests`
- `vendor/bin/phpunit --testsuite=packages` при интеграционном риске

Для `rest`/`rest_doc` дополнительно:
- `vendor/bin/phpunit packages/rest/tests/Unit/SwaggerDocServiceTest.php`
- `vendor/bin/phpunit packages/rest_doc/tests/Feature/SwaggerEndpointsTest.php`
- `vendor/bin/phpunit packages/rest_doc/tests/Feature/SwaggerTryItOutTest.php`
